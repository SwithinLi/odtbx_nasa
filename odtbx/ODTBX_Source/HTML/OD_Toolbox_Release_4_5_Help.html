OD Toolbox 4.5 Release Notes

April 29, 2010

 

The following sections describe the New Features, the Changes to Existing Features, the Known Bugs, and Notes in OD Toolbox Release 4.5 since the prior 4.0 release.

 

NEW FEATURES:

 

Estimator Functions:

 

·         (no significant new estimator features for this release)

 

Measurement Functions:

 

 ·       Added a high-fidelity optical navigation model, opnavmeas, for use in orbit determination simulation and analysis.  opnavmeas generates measurements of landmark positions in the camera focal plane for a given celestial Body, Camera, and Attitude profile by simulating optical image processing and measurement generation for both celestial and landmark-based navigation.

 

 

GPS Measurement Functions and Antenna Link Budget Modeling:

 

A significant amount of new modeling work for GPS measurement generation and GPS antenna Link budget modeling has occurred for this release.  Not only has the ODTBX GPS measurement model, gpsmeas, been upgraded, many new tools for modeling and analyzing the link budget have been added.

 

·         gpsmeas now has the ability to model 2D transmit and receive antennas where antenna gain is based on the elevation and azimuth angles from the antenna boresight.  gpsmeas can now use 2D antenna patterns for either antenna as the user desires.  For 2D receiver patterns, gpsmeas can provide optional receiver antenna orientation states as a function of time as well as a fixed antenna orientation suitable for modeling a terrestrial or orbital receiver antenna.  gpsmeas now models GPS satellite vehicle (SV) yaw angles for 2D transmit patterns.  All of the prior 1D, gain based on boresight elevation angle only, receiver antenna pointing functionality is retained.

 

·         Refactored the physical parameter calculations from gpsmeas into getgpsmeas.  getgpsmeas calculates the underlying physical relationship between a GPS transmitter and receiver such as transmitter and receiver elevation and azimuth angles, range, range rate, GPS SV yaw angle, etc.  getgpsmeas supports gpsmeas and other link budget analysis tools.

 

·         Added alm_pos_az_el, a support routine for GPS link budget calculations in getgpsmeas.  alm_pos_az_el calculates the GPS SV antenna azimuth and elevation angles for a given receiver state.  This also includes a model of the nominal GPS SV yaw angle based on the sun beta angle.  See NOTICES, below.

 

·         Refactored the GPS link budget calculations from gpsmeas into gpslinkbudget.  gpslinkbudget solves the full GPS transmitter-to-receiver link budget for one unknown using the physical parameters of the link budget and the other constant link budget parameters.  It can calculate either:

·         The estimated C/No with both antenna gain patterns,

·         The receive antenna gain with the given C/No and transmitter pattern,

·         Or the transmit antenna gain with the given C/No and receiver pattern.

 

·         A new class, RinexOReader, has been added which reads a RINEX 2.x formatted GPS observation data file.  This class can parse and return data for four different measurement types and is built with user controls to accommodate reading very large files.

 

·         A new tool, rinexo2gpsdata, reads a RINEX 2.x formatted GPS observation data file, via RinexOReader, and converts into one or more MATLAB structures suitable for further processing.  This function also can apply data filters to the GPS measurement data as it is read from the observation file to help with analyzing very large files.  makeGpsData is used to create these structures to hold GPS measurement data.

 

·         A new tool, gps_phys_params, calculates the physical relationship between the GPS transmitter(s) and a receiver given the raw GPS measurement structures from rinexo2gpsdata.  This measurement data is combined with user-provided receiver state data (position, velocity, and optionally attitude, over time) a GPS almanac, and identification data on the GPS transmitter(s) and the receiver into a set of physical parameters for link budget analysis between each transmitter-receiver combination.  As with rinexo2gpsdata, one or more data filters can be applied to the GPS measurement data before processing.  gps_phys_params uses getgpsmeas.  makeGpsTXID is provided for the user to specify information on a GPS SV transmitter such as PRN number or block type.

 

·         A set of classes, all inheriting from FilterGpsData, is provided to filter the raw GPS measurement data.  Filters with various criteria are provided, such as:

·         FilterGpsBlock                               - filter by GPS SV block,

·         FilterGpsCnoThresh                     - filter by raw C/No measurement,

·         FilterGpsExplicitTimes           - filter by explicit GPS time points, with a tolerance,

·         FilterGpsPrn                                    - filter by GPS PRN match,

·         FilterGpsTimeWindow                   - filter by a time window,

·         FilterGpsMulti                               - combines several filters together for use as a single filter.

 

·         A set of classes, all inheriting from FilterPpData, is provided to filter the results of gps_phys_params and associated raw GPS measurement data, if desired.  Filters with various criteria are provided, such as:

·         FilterPpEl                                         - filter by antenna boresight elevation angle (for either transmit or receive antenna),

·         FilterPpExplicitTimes              - filter by explicit UTC time points.

 

·         Two new tools, gps_est_cno and gps_gain, are used to calculate link budget results using the physical parameters from gps_phys_params.  gps_est_cno calculates the estimated C/No from the physical parameters and user-supplied link budget data such as antenna patterns.  gps_gain calculates an antenna gain based on the raw GPS C/No measurement with the physical parameters and user-supplied link budget data when the analyst supplies only one antenna pattern.  gps_gain can be used to calculate either transmit or receive gain values as a function of antenna azimuth and elevation boresight angles.  Both of these tools use gpslinkbudget.

 

·         A new class, FilterEstCnoThresh, is provided to help filter the estimated C/No results from gps_est_cno via a minimum threshold.  This class can also coordinate its filtering of the estimated C/No with filtering the supporting physical parameters data from gps_phys_params, and optionally the associated raw GPS measurements from rinexo2gpsdata.  A class diagram and explanation of all of these filters is included in the ODTBX documentation, accessible from the MATLAB help, and in the help for each filter.

 

·         Plot tools were created to display the data produced or consumed by many of the above tools.  The plot functions are:

·         plot_est_cno                        - Plot the estimated C/No from gps_est_cno vs. time,

·         plot_gps_cno                        - Plot the raw GPS C/No vs. time,

·         plot_gps_gain_2d              - Plot the gain from gps_gain vs. azimuth and elevation angles (2D plots),

·         plot_gps_gain_3d              - Plot the gain from gps_gain and/or antenna gain pattern vs. azimuth and elevation angles (3D plots),

·         plot_gps_pp_azel              - Plot the antenna azimuth and elevation angle physical parameters from gps_phys_params vs. time,

·         plot_gps_pp_gpsyaw         - Plot the modeled GPS yaw angle physical parameter from gps_phys_params vs. time,

·         plot_gps_pp_rrdot            - Plot the range and range rate physical parameters from gps_phys_params vs. time,

·         plot_gps_prange                 - Plot the raw GPS pseudorange vs. time.

 

·         gps_antenna_tools_demo is a demo that illustrates the use of many of the above functions.

 

 

Other Functions:

 

·         Added fgprop, which propagates the position and velocity vector pair over dt using Lagrange's coefficients (f, g, fdot, and gdot), computed using a Newton-Raphson iteration on the universal anomaly.

 

·         Added dcm2q and q2dcm for conversion between quaternions and direction cosine matrices.

 

·         Added slerp for quaternion interpolation.

 

 

CHANGES TO EXISTING FEATURES:

 

Estimator Functions:

 

·         Updated estbat to more properly call the given true datfun in a batch manner, using multiple time points, for the truth simulation.  This change should not affect most results since the design of measurement models should allow them to reproduce consistent results whether called with a single time point or with multiple time points.  However, this is evident when the measurement model does not have consistent behavior.

 

·         Fixed the Variance Sandpiles plots, varpiles, so that they will not have negative y-axis labels for the formal covariance even when zooming or re-scaling the plots.

 

Measurement Functions:

 

·         Function Name Change: The measurement function rrdot is now rrdotang and now includes the ability to calculate the range, range rate, Doppler, and/or angles between two objects.

  

GPS Measurement and Antenna Link Budget Modeling:

 

·         Bugfix in comp_bs_3d for gpsmeas for proper handling of ECEF or ECI inputs for pointing references 5 and 6 (fore/aft, and port/starboard directions relative to geocentric LVLH).

 

·         Bugfix in gpsmeas to fix where the Doppler calculations were actually Range Rate calculations. The user input flag "useDoppler" resulted in Range Rate measurements.  Now gpsmeas correctly calculates the instantaneous Doppler output when requested.  Additionally, the Doppler calculations were corrected to ensure that they occur in an inertial frame regardless of the state vector input a conversion function, Rotation2ECI.

 

·         Added a warning to gpsmeas when using acquisition/tracking thresholds on a single time, as when using a sequential estimator.  These acquisition/tracking thresholds were written with the batch estimator in mind and do not currently have consistent behavior when used with sequential estimators.  The acquisition and tracking thresholds are now defaulted to the same value to ensure consistent default operation on either estimator.

 

·         Bugfix in gpsmeas to properly calculate the measurement Jacobian, H, with respect to the states in the original input frame.  The state vector inputs for gpsmeas can optionally be in any frame, other than ECI, when the user provides a conversion function, Rotation2ECI.

 

 

 

Force Modeling Functions:

 

·         (no significant changes to force models for this release)

 

Other Functions:

 

·         Updated impulsiveBurn, part of estseqs RESTARTRECORD functionality, to allow users to specify both relative and absolute maneuver execution error.

 

·         Updated the installer to warn the user to exit MATLAB before the ODTBX MATLAB path updates can take effect.

 

·         Updated the NASA Directory of Station Locations (NDOSL), in JAT, to more recent data of March 3, 2011.

 

 

 

KNOWN BUGS:

·         Problems with a dual-monitor setup in MATLAB with Windows:

An existing MATLAB bug may affect the way estval and some other ODTBX plots routines display the plots when using dual monitors under Windows.  The MATLAB bug is #301358 (Mathworks account required) and occurs in all recent MATLAB versions, R14 (7.0) through 2011a (7.12) and may affect different graphics drivers differently.

 

·         gpsmeas does not properly apply the acquisition and tracking thresholds when different values are used with a sequential estimator, see above.

 

  

 

NOTES:

·         Other MICE versions: If required, a different version of MICE can be used with MATLAB and ODTBX than the ones supplied with this ODTBX installer.  Frequently this is required when a particular platform is not supported by the ODTBX installer.  MICE and SPICE can be downloaded from http://naif.jpl.nasa.gov/naif/toolkit.html.  Install MICE into a directory under the ODTBX installation directly and name it mice.  The startup.m supplied with ODTBX will look for MICE paths in that directory.  ODTBX paths also expect a mice/kernels subdirectory as a convenient place for SPICE kernels to live.  Create that subdirectory as well, even if you dont intend to use it, so that all of the ODTBX MICE paths will be found.  You will have to download your own SPICE kernels but the pck00009 (or later), the naif0009 (or later), and the de421 kernels are recommended.
